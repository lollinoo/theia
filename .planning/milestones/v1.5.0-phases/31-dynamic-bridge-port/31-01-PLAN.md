<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Theia API (/api/v1/settings) | `bridge_port` value submitted by the browser; must be validated server-side |
| Browser → Bridge binary (localhost:{port}) | Dynamic port is constructed client-side from a settings value; only contacts localhost |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Tampering | PUT /api/v1/settings/bridge_port | mitigate | `numericSettings` check via `strconv.Atoi` rejects non-integer values with 400; frontend also validates 1-65535 range before saving |
| T-31-02 | Elevation of Privilege | Bridge launch URL construction | accept | Port is user-configured and constrained to localhost; the URL contacts `localhost:{port}` only — no remote hosts are reachable via this mechanism; SSRF is not possible because the target is always localhost |
| T-31-03 | Information Disclosure | bridge_port in /api/v1/settings response | accept | The port number is not a secret (it is already exposed in the OS process list); no credentials are leaked |
| T-31-04 | Denial of Service | Out-of-range port values stored in DB | mitigate | Server validates `strconv.Atoi` (non-integer → 400); frontend validates 1-65535 range before issuing PUT; bridge silently fails to connect on wrong port (no server-side impact) |
</threat_model>
