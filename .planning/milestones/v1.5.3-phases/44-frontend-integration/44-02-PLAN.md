<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form input -> REST handler | User-provided override values cross into backend validation and persistence |
| REST resource -> frontend parser | Backend device attributes are trusted only after explicit type parsing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-05 | Tampering | `PUT /api/v1/devices/{id}` poll override input | mitigate | Validate `poll_interval_override` in the handler and accept only `null` or integer seconds in the `5..3600` range. Severity: HIGH. |
| T-44-06 | Denial of Service | scheduler cadence input | mitigate | Reject zero, negative, and huge override values before they can reach persistence or runtime scheduling. Severity: HIGH. |
| T-44-07 | Repudiation | tri-state override updates | mitigate | Use explicit keep/clear/set semantics in `DeviceUpdate` and cover them with handler and service tests so override changes are deterministic. Severity: MEDIUM. |
| T-44-08 | Information Disclosure | frontend device parsing | accept | `poll_class` and `poll_interval_override` are operational metadata already safe to expose in the device resource. Severity: LOW. |

Security gate status: ASVS L1, block_on=high. The phase blocks on Tasks 1-2 because invalid override values would be a HIGH-severity tampering/DoS issue if left unvalidated.
</threat_model>
