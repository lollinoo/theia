<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Device source -> scheduler refresh | DB/cache-fed device rows determine which tasks exist and what cadence each one uses |
| Refresh reconciliation -> heap state | Removing or mutating scheduled keys must not corrupt future dispatch ordering |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-05 | Denial of Service | heap state | mitigate | The scheduler uses one heap item per `device + volatility` key and refresh updates existing keys in place instead of pushing duplicates, preventing unbounded heap growth across refresh cycles. Severity: HIGH if unmitigated. |
| T-41-06 | Tampering | device refresh reconciliation | mitigate | Unmanaged and deleted devices are removed from future scheduling immediately, or marked `disabled` if already in flight, so stale rows cannot keep generating future polls. Severity: MEDIUM. |
| T-41-07 | Denial of Service | pull-based refresh lag | accept | Periodic refresh is a deliberate Phase 41 decision; a newly added device may wait until the next refresh cycle before getting scheduled. This is bounded by the 30-second default and matches the locked design. Severity: LOW. |
| T-41-08 | Repudiation | due-time mutation on refresh | mitigate | Existing items keep their current `dueAt`; refresh only updates metadata and intervals, making the reconciliation behavior deterministic and testable. Severity: MEDIUM. |

Security gate status: ASVS L1, block_on=high. The duplicate-heap-growth risk is mitigated directly in this plan.
</threat_model>
