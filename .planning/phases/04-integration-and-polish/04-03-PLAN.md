<!-- Existing backend API contracts -->

Settings API:
- GET /api/v1/settings -> {data: Record<string, string>}
- PUT /api/v1/settings/{key} with body {value: string} -> {data: {key: value}}

Device API:
- POST /api/v1/devices with body {hostname, ip, snmp_community, snmp_version, ...} -> device resource
- PUT /api/v1/devices/{id} with body {fields to update} -> device resource
- DELETE /api/v1/devices/{id} -> 204

Setting keys from internal/domain/settings.go:
```go
const (
  SettingPrometheusURL      = "prometheus_url"
  SettingGrafanaURL         = "grafana_url"
  SettingPollingInterval    = "polling_interval_seconds"
  SettingSNMPWorkerPoolSize = "snmp_worker_pool_size"
  SettingSNMPTimeout        = "snmp_timeout_seconds"
  SettingSNMPRetries        = "snmp_retries"
)
```

Per-device polling override: The backend does not yet have a per-device polling override mechanism. The global `polling_interval_seconds` setting is read by the Poller each cycle. For per-device overrides, use a settings key convention: `polling_interval_seconds:{device_id}` which the collector/poller can read. The existing SettingsRepository.Get/Set works for arbitrary keys. The Poller will need a minor update to check per-device keys (deferred to plan 04 or handled as a backend addition in this plan if feasible).

From 04-CONTEXT.md decisions:
- Preset dropdown: 15s, 30s, 60s, 120s, 300s plus 'Custom...' for arbitrary values
- Changes auto-save with debounce (~500ms)
- Add device form: hostname/IP, SNMP community, SNMP version (v2c/v3), optional display name
- Device config: polling override, custom Grafana URL, edit device properties, delete with confirmation

**client.ts additions:**
- `fetchSettings(): Promise<Record<string, string>>` -- GET /api/v1/settings, return `data` field
- `updateSetting(key: string, value: string): Promise<void>` -- PUT /api/v1/settings/{key} with {value}
- `createDevice(payload: {hostname: string, ip: string, snmp_community: string, snmp_version: string}): Promise<Device>` -- POST /api/v1/devices
- `updateDevice(id: string, payload: Partial<{hostname: string, ip: string, snmp_community: string}>): Promise<Device>` -- PUT /api/v1/devices/{id}
- `deleteDevice(id: string): Promise<void>` -- DELETE /api/v1/devices/{id}

Note: If `fetchSettings` was already added in plan 02, reuse it. Do not duplicate.

**SettingsPanel.tsx:** Global settings panel content.
- On mount, fetch current settings via `fetchSettings()`
- Display polling interval as a preset dropdown: 15s, 30s, 60s (default), 120s, 300s, Custom
- When "Custom" is selected, show a numeric input for arbitrary seconds value (min 5, max 3600)
- On change, debounce 500ms then call `updateSetting('polling_interval_seconds', value)`
- Show a subtle "Saved" indicator that fades after 2 seconds on successful save
- Also display Grafana URL and Prometheus URL as editable text inputs with the same debounced auto-save pattern
- Dark themed, matching app style. Use labels with text-text-secondary, inputs with bg-bg-elevated border-border-subtle

**AddDevicePanel.tsx:** Add device form in the side panel.
- Fields: Hostname/IP (required text input), SNMP Community (required text input, default "public"), SNMP Version (dropdown: v2c (default), v3), Display Name (optional text input)
- Submit button at bottom: "Add Device"
- On submit: validate hostname/IP is not empty, call `createDevice()`, on success close the panel and trigger a topology reload
- On error: show error message inline below the form
- Provide an `onDeviceAdded` callback prop so Canvas can reload topology after a device is added
- Loading state on submit button
