    - when a due item is already queued, another due event does not enqueue a duplicate
    - when a due item is in-flight, another due event sets `pending=true` and nothing more
    - `Complete()` for a non-disabled item with `pending=true` queues exactly one immediate rerun and clears `pending`
    - `Complete()` for a non-disabled item with no pending rerun reinserts the item into the heap at `jitteredNext(finishedAt, interval, s.rnd)`
    - `Complete()` for a disabled item drops it permanently and does not reinsert it
    - completion timestamp fallback: if `c.FinishedAt.IsZero()`, use `s.now().UTC()`
  </behavior>
  <action>
    Extend `internal/scheduler/scheduler.go` with:

    ```go
    func (s *Scheduler) handleCompletion(c Completion)
    ```

    Implement these exact rules:
    - lookup `item := s.items[c.Key]`; if missing, ignore completion
    - decrement `s.inFlight` only when `item.inFlight` was true
    - clear `item.inFlight`
    - if `item.disabled`:
      - delete from `s.items`
      - do not reinsert
      - return
    - if `item.pending`:
      - clear `item.pending`
      - call `enqueueReady(item)` exactly once
      - set `item.task.DueAt` to `finishedAt`
      - do not push the item back into the heap yet
      - return
    - otherwise:
      - compute `next := jitteredNext(finishedAt, item.interval, s.rnd)`
      - write `item.dueAt = next`
      - write `item.task.DueAt = next`
      - push back into the heap

      - pop the item
      - if `item.disabled`, delete from `items` and continue
      - if `item.inFlight`, set `item.pending = true` and continue
      - if `item.queued`, continue
      - otherwise call `enqueueReady(item)`

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-09 | Denial of Service | simultaneous SNMP work | mitigate | `maxInFlight()` caps global dispatch using the existing `snmp_worker_pool_size` setting with safe fallback `5`; scheduler never dispatches beyond the limit. Severity: HIGH if unmitigated. |
| T-41-10 | Denial of Service | backlog growth | mitigate | Extra due firings for an already queued/in-flight key collapse to one pending rerun, preventing unbounded ready-queue or heap growth. Severity: HIGH if unmitigated. |
| T-41-11 | Tampering | disabled task reinsertion | mitigate | Removed/unmanaged keys marked `disabled` are dropped on completion and never reinserted, preventing stale device rows from continuing to drive polls. Severity: MEDIUM. |
| T-41-12 | Repudiation | completion timing drift | mitigate | Reinsertion uses the explicit completion timestamp, or `s.now()` when absent, so cadence advance is deterministic and testable instead of inferred from stale due times. Severity: MEDIUM. |

Security gate status: ASVS L1, block_on=high. Both high-severity scheduler risks (unbounded concurrency and unbounded backlog) are directly mitigated in this plan.
