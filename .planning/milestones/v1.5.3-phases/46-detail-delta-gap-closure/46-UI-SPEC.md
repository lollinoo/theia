## Visual Hierarchy

- The open side panel remains the only user-visible indicator that device detail mode is active.
- In device-scoped interface panels, the panel title is the first read, each interface card header is the second read, and TX/RX/utilization values are the third read.
- Link-scoped interface panels keep the existing negotiation summary above the two endpoint cards; Phase 46 does not collapse or redesign that layout.
- Live throughput updates change numeric values and utilization bar fill only. They must not reorder cards, animate card chrome, or change drawer/header styling.

## Interaction Contract

- Subscription ownership stays unchanged from Phase 43. The canvas `SidePanel` remains the only owner of detail subscriptions. Device-scoped `interfaceStats` continues to subscribe by device ID, and `deviceConfig` remains eligible. Link-scoped `interfaceStats`, `link-details`, alerts, settings, dashboard/table surfaces, and generic selection remain unsubscribed (source: `43-UI-SPEC.md`, `frontend/src/components/canvas/detailSubscription.ts`).
- The targeted detail path may update only the selected device's `device_metrics`, `device_statuses`, and `link_metrics[device.id]` needed by the device-scoped interface panel. It must not widen to other devices' `link_metrics`, alerts, hostnames, models, topology, or a second websocket message family (source: Phase 46 roadmap goal, milestone audit INT-03, `frontend/src/types/metrics.ts`).
- Freshness promise: when `Per-Interface Stats` is open for a device and the WebSocket is connected, that panel's TX, RX, and utilization values refresh from the targeted `snapshot_delta` sent immediately after the selected device's performance poll completes. The user must not wait for the next overview broadcast tick to see new throughput (source: Phase 46 success criteria, `internal/worker/pipeline.go`).
- "Relevant poll" for throughput means the selected device's performance poll. Operational or static polls may still send same-device detail metadata, but they do not promise changed TX/RX/utilization values and must not imply that interface throughput was refreshed.
- The frontend continues merging targeted detail into the single shared `snapshot` atom via `mergeSnapshotDelta`. No panel-local WebSocket, second cache, or REST polling loop is added. Device interface rows continue reading throughput from `snapshot.link_metrics[device.id]` (source: `frontend/src/hooks/useWebSocket.ts`, `frontend/src/types/metrics.ts`, `InterfaceStatsPanel.tsx`).
- Interface inventory, ordering, names, descriptions, speed, and `oper_status` remain REST-backed from `fetchDeviceInterfaces(device.id)` on panel open. Phase 46 does not make those fields live-updating over targeted detail (source: `InterfaceStatsPanel.tsx`).
- Link-scoped `Per-Interface Stats` panels remain overview-driven and REST-backed on both ends. They do not subscribe by device or link ID and continue refreshing throughput only when the regular overview snapshot changes (source: `frontend/src/components/canvas/detailSubscription.ts`, `CanvasPanels.tsx`).
- Unsubscribed clients remain on the existing overview snapshot/delta contract. Opening a device interface panel in one client must not change another client's payload shape or expose device-scoped link metrics outside the subscribed path (source: Phase 46 success criteria, milestone audit INT-03).
- Close, switch, and reconnect behavior stays unchanged from Phase 43. Closing the drawer or switching away unsubscribes immediately; reconnect re-sends the active device subscription automatically if the same device panel is still open (source: `43-UI-SPEC.md`, `frontend/src/hooks/useWebSocket.ts`).
- Do not add a "live", "subscribed", or "updated just now" badge, toast, pulse, skeleton, or drawer color shift for this fix. Existing reconnect chrome and in-card `--` placeholders remain the only degraded-state cues.
