## Visual Hierarchy

- Selected node is the canvas focal point whenever device detail is active: keep the accent glow/ring on the selected node only, and keep neighboring nodes on neutral surfaces so the eye lands on the active device first.
- Open detail drawer is the primary information plane after selection: the header title row carries the strongest text emphasis, the close affordance is secondary, and body sections step down into 20px headings, 12px labels, and 14px values.
- Full-canvas empty and error states collapse the canvas into one centered card on the dominant background: use the 28px display heading as the first read, 14px supporting copy as the second read, and any retry/recovery action as the only accent-treated element.

## Interaction Contract

- Subscription owner: the existing canvas right-side `SidePanel` remains the only owner of detail subscriptions. Only device-scoped panel content may subscribe: `deviceConfig` and device-only `interfaceStats`. Link details, alerts, settings, dashboard surfaces, generic selection, and search do not participate in Phase 43 (source: `43-CONTEXT.md` D-03 to D-05, `CanvasPanels.tsx`).
- Open behavior: opening a device-scoped canvas panel sends exactly one `subscribe_detail` message for that device ID over the existing WebSocket connection. Do not open a second socket, do not add REST polling, and do not trigger extra backend polling cadence changes (source: `43-CONTEXT.md` D-06 to D-07, `useWebSocket.ts`).
- Close and switch behavior: closing the drawer with the header close button, `Escape`, pane click, or replacing one device-scoped panel with another must send `unsubscribe_detail` for the previous device immediately before any replacement subscribe. One client connection maps to at most one active detail device at a time (source: `43-CONTEXT.md` D-04, `SidePanel.tsx`, `Canvas.tsx`).
- Rendering model: detail updates must arrive as additive `snapshot_delta` payloads and merge into the existing single `snapshot` atom. No `device_detail` message family, no panel-local mirror state, and no separate selected-device cache are allowed (source: `43-CONTEXT.md` D-01 to D-02, Pitfall 8, `frontend/src/types/metrics.ts`).
- Visible chrome: Phase 43 adds no new badge, toast, spinner, drawer, or color treatment to announce “detail mode.” The open side panel is the only user-visible indicator that detail delivery is active. Reuse the current right-anchored `320px` drawer, header title, and close affordance unchanged (source: phase boundary in `43-CONTEXT.md`, `SidePanel.tsx`).
- Initial content: when a device panel opens, render immediately from whatever data already exists in the shared snapshot. Do not blank the panel waiting for the next poll; the next normal device poll may then refresh the same fields via targeted delta (source: existing snapshot-first UI model in `useWebSocket.ts`, `CanvasPanels.tsx`).
- Reconnect behavior: if the WebSocket reconnects while the same device panel is still open, automatically re-send `subscribe_detail` after `onopen`. Reuse the existing reconnect banner and recovery behavior; do not add panel-specific error chrome for transient disconnects (source: `useWebSocket.ts`, `ReconnectBanner.tsx`, default required by existing reconnect model).
- Scope boundary: do not surface new health/freshness/polling badges, panel rows, or metric explanations in Phase 43. This phase wires delivery only; Phase 44 owns presentation of those backend-computed fields (source: `43-CONTEXT.md` D-08 to D-09, ROADMAP Phase 44 boundary).
