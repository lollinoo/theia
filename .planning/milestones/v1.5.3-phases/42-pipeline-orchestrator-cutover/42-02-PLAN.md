<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Static collector result -> persistence helper | Discovery output becomes persisted device/interface/link topology |
| Persistence helper -> caller-owned notify channel | The helper may discover topology changes, but the caller controls when the WebSocket event is emitted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-05 | Tampering | duplicated persistence logic | mitigate | A single `ApplyStaticDiscovery` helper removes divergent topology-write paths between `probeDevice()` and the new pipeline. Severity: HIGH if unmitigated because one path can drift and silently stop persisting topology updates. |
| T-42-06 | Denial of Service | topology notify ordering | mitigate | The helper never emits directly to `TopologyNotify`; callers keep the existing non-blocking send and the broadcast layer drains it after snapshot/delta. Severity: HIGH if unmitigated because out-of-order events can force frontend split-brain reloads. |
| T-42-07 | Repudiation | discovered topology changes | mitigate | Tests assert that link creation and interface mutation both surface as `TopologyChanged`, making the broadcast-trigger contract explicit. Severity: MEDIUM. |
| T-42-08 | Information Disclosure | persistence helper inputs | accept | Inputs stay inside `internal/service` and are not exposed via HTTP or WebSocket surfaces. Severity: LOW. |

Security gate status: ASVS L1, block_on=high. No HIGH-severity issue remains once helper reuse and notify-order tests pass.
</threat_model>
