# Requirements: Theia v1.3.0 Frontend Redesign

**Defined:** 2026-03-25
**Core Value:** Network operators can see their entire topology at a glance with live stats on every device and link

## v1.3.0 Requirements

Requirements for the frontend redesign milestone. Each maps to roadmap phases.

### Design Foundation

- [x] **FOUND-01**: CSS variable token system defines semantic color tokens (surface, text, status, accent) mapped to primitive values per theme
- [x] **FOUND-02**: Tailwind CSS v3 migrated to v4 using `@theme` directive for native CSS variable integration
- [x] **FOUND-03**: React Flow upgraded from v11 (`reactflow`) to v12 (`@xyflow/react`) with native `colorMode` support
- [x] **FOUND-04**: Outfit font loaded via self-hosted Fontsource variable package for display and UI text
- [x] **FOUND-05**: JetBrains Mono font loaded via self-hosted Fontsource variable package for technical readouts and metric values
- [x] **FOUND-06**: All hardcoded hex color values across 25+ component files replaced with CSS variable token references

### Theming

- [x] **THEME-01**: User can toggle between dark and light themes via a UI control
- [x] **THEME-02**: User's theme preference persists across browser sessions via localStorage
- [x] **THEME-03**: App defaults to OS dark/light preference when no explicit user choice exists
- [x] **THEME-04**: No flash of wrong theme (FOWT) on page load — inline script in `<head>` applies theme before render
- [x] **THEME-05**: All 25+ components are readable and visually correct in both dark and light themes

### Component Restyling

- [x] **COMP-01**: DeviceCard restyled with Neon Topography aesthetics — glow status nodes, Outfit labels, JetBrains Mono values, no-line internal sectioning
- [x] **COMP-02**: Context menu restyled with Material Symbols icons, separator support between action groups, glassmorphism surface, and danger/destructive action styling
- [x] **COMP-03**: NavBar restyled to Neon Topography design with updated branding and navigation structure
- [x] **COMP-04**: SidePanel, Toolbar, and ZoomControls restyled with surface hierarchy and ghost border fallbacks
- [x] **COMP-05**: SettingsPanel and all sub-panels (SNMP, SSH, Vendor) restyled with Neon Topography aesthetics
- [x] **COMP-06**: AlertsPanel and SearchOverlay restyled with glassmorphism and glow effects where appropriate
- [x] **COMP-07**: Dashboard, DeviceTable, and DeviceRow restyled with new typography and color tokens
- [x] **COMP-08**: AddDevicePanel, DeviceConfigPanel, and LinkCreatePanel restyled with updated form styling
- [x] **COMP-09**: LinkDetailsPanel and InterfaceStatsPanel restyled with JetBrains Mono for metric values
- [x] **COMP-10**: LinkEdge connections restyled with updated colors and hover effects matching theme
- [x] **COMP-11**: Bloom/radial-blur effects applied behind status-critical elements (area cards, status indicators)
- [x] **COMP-12**: "No-line rule" enforced — layout regions use surface color tiers for depth, not 1px borders

### Area Backend

- [x] **AREA-01**: Database schema includes `areas` table with id, name, description, and accent color fields
- [x] **AREA-02**: REST API endpoints for area CRUD (`/api/v1/areas` — GET, POST, PUT, DELETE)
- [x] **AREA-03**: Devices table has nullable `area_id` foreign key referencing the areas table
- [x] **AREA-04**: REST API supports assigning/unassigning a device to an area via device update endpoint

### Area Frontend

- [x] **AREA-05**: User can create, edit, and delete areas in the Settings panel
- [x] **AREA-06**: User can assign a device to an area via a dropdown in DeviceConfigPanel
- [x] **AREA-07**: OSPF Area Hub view displays aggregate stats (network uptime, aggregate health) and per-area cards
- [x] **AREA-08**: Per-area cards show area name, description, health status, device count, and active link count with area-specific accent colors
- [x] **AREA-09**: Floating navigation pill allows switching between Global view and individual area views
- [x] **AREA-10**: Atmospheric watermark displays contextual text (e.g., "GLOBAL TOPOLOGY") that changes per selected area
- [x] **AREA-11**: Area-filtered topology canvas shows only devices and links belonging to the selected area when an area is active in the nav pill

## Out of Scope

| Feature | Reason |
|---------|--------|
| Custom user-defined theme colors | Design system is opinionated; custom colors would break curated palette and bloom effects |
| Per-device custom colors/icons | Creates visual noise, undermines status glow hierarchy |
| Animated canvas backgrounds | Competes with data for attention, wastes GPU at 100+ nodes |
| Drag-to-assign devices to areas on canvas | Conflates spatial position with logical grouping |
| SNMP-derived OSPF area data | SNMP exporters don't expose area OIDs; manual grouping is simpler |
| Third theme or time-of-day auto-blend | OS-level auto dark mode handles this; three theme sets too costly |
| Inline area editing on Hub page | Conflicting click targets (navigate vs edit); editing belongs in Settings |
| Route count display in areas | Too expensive to fetch at scale with many routes present |
