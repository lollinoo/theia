# Milestone v1.5.8: Live Refresh Hardening

**Status:** ✅ SHIPPED 2026-04-19
**Phases:** 1-4
**Total Plans:** 11

## Overview

This milestone hardened Theia's live-refresh path without replacing the core architecture. The work started by making the refresh path measurable, then bounded backend failure modes, then removed unnecessary topology work from the frontend, and finally validated the result at the target scale of 300 devices on PostgreSQL in the real Docker stack.

## Phases

### Phase 1: Refresh Path Instrumentation

**Goal**: The live-refresh path exposes enough backend and frontend measurements to show where time, overflow, and topology reload work are occurring
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:

- [x] 01-01: Add backend refresh-path metrics for snapshots, scheduler lag, and reload reasons
- [x] 01-02: Add overflow and backpressure accounting to state propagation and WebSocket broadcast paths
- [x] 01-03: Add frontend or harness-level measurement hooks for canvas rebuild and layout cost

**Details:**
- Requirements: `OBSV-01`, `OBSV-02`, `OBSV-03`, `OBSV-04`
- Success criteria:
  1. Backend metrics expose snapshot construction cost and scheduler lag for the live-refresh pipeline
  2. Backend metrics expose overflow or drop counters for runtime state propagation and WebSocket broadcast paths
  3. The system can attribute topology reloads by cause instead of only logging generic refresh activity
  4. The frontend or harness can measure canvas rebuild and layout cost during live-refresh scenarios

### Phase 2: Backend Live-State Resilience

**Goal**: The backend keeps live state fresh under slow Prometheus responses and backpressure by failing fast and recovering explicitly
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: Harden the Prometheus client and query call sites with explicit timeout policy
- [x] 02-02: Implement overflow accounting and explicit resync semantics in state-store and WebSocket delivery
- [x] 02-03: Add targeted tests for slow Prometheus, reconnect storms, and slow-client backpressure recovery

**Details:**
- Requirements: `LIVE-02`, `LIVE-03`, `PROM-01`, `PROM-02`, `PROM-03`
- Success criteria:
  1. Prometheus runtime queries always run with a shared HTTP timeout and an operation-level context timeout
  2. Prometheus timeout, error, and latency behavior is visible in observability metrics
  3. Overflow in runtime state propagation or WebSocket broadcast marks the stream for explicit resync instead of silent drift
  4. Reconnect or overflow recovery can drive a trustworthy full snapshot/resync path

### Phase 3: Frontend Incremental Reconciliation

**Goal**: Runtime-only updates change the operator view without triggering structural graph reloads, layout churn, or unnecessary viewport resets
**Depends on**: Phase 2
**Plans**: 3 plans

Plans:

- [x] 03-01: Separate topology loading from runtime overlay reconciliation in canvas data flow
- [x] 03-02: Coalesce reconnect and topology events while preserving stable graph identity
- [x] 03-03: Add regression tests for runtime-only updates, saved-position cases, and reconnect behavior

**Details:**
- Requirements: `TOPO-01`, `TOPO-02`, `TOPO-03`, `LIVE-01`, `LIVE-04`
- UI hint: yes
- Success criteria:
  1. Runtime-only health, metrics, alerts, and reachability updates do not trigger a full topology reload
  2. Existing node and edge identity is preserved during runtime-only reconciliation
  3. Force-layout and viewport refit are skipped when saved positions remain valid and topology is unchanged
  4. Reconnect and topology-change events are coalesced so the browser avoids redundant structural refresh work

### Phase 4: Scale Validation And Hardening

**Goal**: The milestone is proven on the target operating environment and the highest-value issues discovered during testing are closed
**Depends on**: Phase 3

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: Add refresh-path observability before changing backend or frontend behavior. Rationale: later tuning needed hard evidence rather than guesswork.
- Decision: Bound Prometheus runtime work with a shared client timeout plus operation deadlines. Rationale: live refresh latency needed explicit failure limits.
- Decision: Turn shared overflow into explicit `resync_required` plus full snapshot recovery. Rationale: silent drift was less acceptable than narrow, visible recovery.
- Decision: Rebuild topology only on structural identity change and coalesce reconnect or topology events through one short refresh window. Rationale: runtime-only updates needed to stay off the expensive structural path.
- Decision: Treat validation gaps as evidence problems first and code problems only when a concrete seam defect reproduces. Rationale: the final Phase 4 reruns did not justify extra refresh-path code changes.

**Issues Resolved:**

- Missing backend and browser-visible refresh-path metrics
- Unbounded Prometheus runtime latency and weak runtime request observability
- Silent shared-overflow drift in runtime propagation and WebSocket delivery
- Runtime-only updates triggering unnecessary topology reload and layout churn
- Lack of repeatable 300-device PostgreSQL validation evidence

**Issues Deferred:**

- Phases 1-3 remain Nyquist-partial and need follow-up validation closure
- `SCAL-02` browser proof should be paired to one exact successful `metrics.prom` artifact
- `SCAL-03` would benefit from dedicated live reconnect-storm or slow-client fault injection

**Technical Debt Incurred:**

- Closeout still relies partly on manually correlated validation evidence
- Target-scale storm proof is still partly regression-led instead of fully live-harness-driven
- The next milestone still needs fresh requirements before new roadmap phases can be planned
