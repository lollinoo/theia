<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SNMP device -> operational collector | Uptime and interface-status data from the network device becomes user-visible operational state |
| SNMP device -> static collector | Discovery data (hostname, vendor, topology neighbors) flows into typed results for later pipeline consumers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-09 | Tampering | Missing/unsupported operational OIDs treated as hard failure | mitigate | `PollOperationalStatus` falls back to standard OIDs and preserves partial results for empty walks or missing values; only transport/query errors set `Err`. Severity: MEDIUM. |
| T-40-10 | Denial of Service | Static collector accidentally reintroducing DB-backed reprobe writes | mitigate | `StaticCollector` is a pure wrapper around `snmp.DiscoverDevice()` and has no DeviceService/DeviceRepo collaborators. Severity: MEDIUM. |
| T-40-11 | Information Disclosure | Static discovery result includes hostname/model/vendor data | accept | The result stays inside `internal/collector` for later in-process pipeline consumption and does not widen any HTTP or WS contract in this phase. Severity: LOW. |
| T-40-12 | Tampering | Vendor-specific operational OID override drift | mitigate | Operational collector resolves OIDs exclusively through `Registry.ResolveOperationalOIDs`, keeping a single source of truth shared with vendor config tests. Severity: MEDIUM. |

Security gate status: ASVS L1, block_on=high. No HIGH-severity threat remains open in this plan.
</threat_model>
