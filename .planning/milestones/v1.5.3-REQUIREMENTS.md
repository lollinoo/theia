# Requirements: MikroTik Theia

**Defined:** 2026-04-11
**Core Value:** Network operators can see their entire topology at a glance with live stats on every device and link, drill into Grafana for deep dives, and manage devices directly -- all from a single interactive map.

## v1.5.3 Requirements

Requirements for the SNMP Pipeline Architecture milestone. Each maps to roadmap phases.

### State Engine

- [x] **STATE-01**: Backend maintains a centralized in-memory device state store that holds per-device status, metrics, health, and staleness -- replacing DeviceLinkCache as single source of truth
- [x] **STATE-02**: Backend computes health state (healthy/warning/critical/unknown) from metrics and thresholds -- frontend receives a single enum per device
- [x] **STATE-03**: Backend applies hardcoded threshold defaults with hysteresis (CPU warn 70%/clear 60%, critical 90%/clear 80%; same pattern for memory, temperature) to prevent flapping
- [x] **STATE-04**: Backend implements soft/hard state transitions -- 1 failure = soft-down (degraded), 3 consecutive failures = hard-down; single success = immediate recovery
- [x] **STATE-05**: State engine computes diffs on each update and emits only meaningful changes to the existing FNV-64a delta WS broadcast layer

### Polling Architecture

- [x] **POLL-01**: Discovery walks (ifTable, ifXTable, LLDP, CDP) run on a slower shared static cadence (300s system-wide in v1.5.3) separate from metrics polling
- [x] **POLL-02**: Performance metrics polling (CPU, memory, temperature, counters) runs per device on the poll-class cadence (core=30s, standard=60s, low=300s, plus performance-only override) independent of discovery
- [x] **POLL-03**: Vendor config defines OID volatility classes (static/operational/performance) per OID group
- [x] **POLL-04**: Scheduler distributes polls across the interval window with deterministic FNV-based per-device offset and concurrency limits -- no thundering herd bursts
- [x] **POLL-05**: Backend auto-classifies devices into the 3-bucket polling model (core=30s for router/switch, standard=60s for AP/unknown, low=300s for virtual), with per-device performance-tier overrides available separately
- [x] **POLL-06**: User can override polling frequency per device via API and UI

### Data Pipeline

- [x] **PIPE-01**: SNMP is the primary data source for all managed devices; Prometheus becomes optional enrichment
- [x] **PIPE-02**: Stateless collectors exist per volatility class (PerformanceCollector, OperationalCollector, StaticCollector) wrapping existing SNMP/Prometheus clients
- [x] **PIPE-03**: Pipeline orchestrator replaces Poller + MetricsCollector, wiring scheduler to workers to state engine to WS broadcast
- [x] **PIPE-04**: Counter rate computation detects resets (new < old = discard), enforces sanity bounds (rate > 110% link speed = discard), and skips rate on first poll after a gap

### WebSocket & Frontend

- [x] **WS-01**: WS payload includes per-device last_polled_at timestamp; frontend displays freshness as fresh/stale/dead tiers
- [x] **WS-02**: User can click a device on the canvas to subscribe to higher-frequency detail updates via WS; deselecting unsubscribes
- [x] **WS-03**: Frontend displays backend-computed health state as colored status indicators on device cards
- [x] **WS-04**: Frontend shows per-device polling interval label (e.g., "Polling every 15s")

## Future Requirements

Deferred to future release. Tracked but not in current roadmap.

### Thresholds & Configuration

- **THRESH-01**: User can configure threshold values (warning/critical %) per metric via API and UI
- **THRESH-02**: User can apply threshold profiles (templates) to device groups

### Advanced Polling

- **ADVPOLL-01**: User can trigger discovery-on-demand for a specific device via API
- **ADVPOLL-02**: Backend computes per-interface health state for link-level detail

### Infrastructure

- **INFRA-01**: SNMP trap receiver for instant event notification without polling
- **INFRA-02**: Distributed polling across multiple poller nodes
- **INFRA-03**: Formal flapping detection (Nagios-style percent state change)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Configurable threshold profiles | Zabbix's most complex subsystem; ship hardcoded defaults now |
| SNMP trap receiver | Fundamentally different data flow, separate milestone |
| Distributed polling | Requires PostgreSQL migration, different state management |
| Per-device patch WS delivery | Existing FNV section-hash delta is sufficient and well-tested |
| Metric history/time-series storage | That's what Prometheus + Grafana are for |
| Real-time sub-second polling | SNMP not designed for it; minimum 10s interval |
| Flapping detection | Soft/hard state + hysteresis covers 95% of cases |
