<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SNMP probe -> service | Untrusted device-reported sysObjectID/sysDescr crosses into DeviceService.probeDevice and influences DeviceType -> PollClass classification |
| Operator -> API | Operator-supplied poll_interval_override (eventually wired in Phase 40+) crosses into DeviceService and gates the auto-reclassify guard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-11 | Tampering | probeDevice() classification | accept | A malicious SNMP responder can lie about sysObjectID and trick detection into returning a different DeviceType (e.g., claim to be a router when it's a virtual host). Worst case: device gets polled at 30s instead of 300s -- a 10x amplification of poll traffic toward that single device. Already a pre-existing risk of the SNMP detector; this plan does not enlarge the attack surface. Accept. |
| T-39-12 | Denial of Service | Reclassify guard | mitigate | A device with an unset PollIntervalOverride that flaps device_type between router (core=30s) and virtual (low=300s) on every probe would cause the scheduler (Phase 41+) to constantly re-bucket it. Mitigation: the guard is read-only on PollIntervalOverride and simply assigns the new PollClass; no cascading writes, no external I/O, single in-process map operation per probe. Bounded by existing probe interval. |
| T-39-13 | Elevation of Privilege | Override bypass | mitigate | An attacker who can write to a device row (e.g., through a future API bug) could set PollIntervalOverride to a tiny value to weaponize the polling loop as a DoS amplifier. Mitigation belongs in Phase 40/42 when the API actually accepts override writes (per 39-CONTEXT deferred items: "API validation for poll_interval_override -- minimum >=10s"). This plan only stores and reads the value; no API surface added here. Tracked as deferred. |
| T-39-14 | Information Disclosure | ROADMAP edit | accept | ROADMAP.md is a tracked planning artifact; changes are visible in git history. No secrets touched. Accept. |
</threat_model>
