# Requirements Archive: v1.5.8 — Live Refresh Hardening

**Archived:** 2026-04-19
**Outcome:** 17/17 milestone requirements completed
**Carry-Forward Candidates:** `ARCH-01`, `ARCH-02`, `PROD-01`, `PROD-02`

## Outcome Summary

- Validated in this milestone: all 17 v1 live-refresh hardening requirements
- Adjusted during execution: `SCAL-02` and `SCAL-03` closed through Phase 4 evidence completion rather than additional refresh-path code changes
- Dropped from this milestone: none
- Deferred to the next milestone definition: architecture follow-ups and broader product expansion candidates listed below

## Validated Requirements

### Topology Reconciliation

- [x] **TOPO-01**: The system reloads topology entities only when nodes or links materially change
- [x] **TOPO-02**: The frontend preserves existing node and edge identity during runtime-only updates
- [x] **TOPO-03**: The frontend skips force-layout and viewport refit when saved positions are valid and topology is unchanged

### Runtime Delivery

- [x] **LIVE-01**: Operators receive live health, metrics, alerts, and reachability updates without triggering a full topology reload
- [x] **LIVE-02**: The backend detects state-store or WebSocket backpressure overflow and marks the stream for explicit resync
- [x] **LIVE-03**: Overflow or reconnect recovery delivers a full snapshot/resync path instead of relying on implicit eventual recovery
- [x] **LIVE-04**: Reconnect and topology change events are coalesced so the frontend does not perform redundant structural refresh work

### Prometheus Resilience

- [x] **PROM-01**: Every Prometheus runtime query uses a shared HTTP client with a hard timeout
- [x] **PROM-02**: Every Prometheus runtime query applies an operation-level context timeout and releases cancellation resources correctly
- [x] **PROM-03**: Prometheus timeout, error, and latency behavior is exposed through backend observability metrics

### Observability

- [x] **OBSV-01**: The backend exposes snapshot construction duration and scheduler lag metrics for the live refresh path
- [x] **OBSV-02**: The backend exposes overflow/drop counters for runtime state propagation and WebSocket broadcast paths
- [x] **OBSV-03**: The system records topology reload counts by reason, including reconnect-driven and structural-change-driven refreshes
- [x] **OBSV-04**: The frontend or test harness can measure canvas rebuild and layout cost during live refresh scenarios

### Scale Validation

- [x] **SCAL-01**: The system passes targeted validation for 300 monitored devices on PostgreSQL in the real Docker stack
- [x] **SCAL-02**: Target-scale validation proves that runtime-only updates do not trigger full topology reload behavior
- [x] **SCAL-03**: Target-scale validation covers slow Prometheus responses, reconnect storms, topology-change storms, and slow-client/backpressure scenarios

## Carry-Forward Candidates

### Architecture Follow-ups

- **ARCH-01**: Extract additional canvas and pipeline submodules beyond those strictly required for the milestone
- **ARCH-02**: Introduce finer-grained topology entity diffs beyond the milestone resync strategy

### Product Expansion

- **PROD-01**: Expand live-refresh support to more vendors beyond the current MikroTik focus
- **PROD-02**: Revisit transport alternatives beyond the current WebSocket contract if the hardened model is still insufficient at higher scale

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full rewrite of the canvas architecture | Too risky for the milestone; current architecture should be hardened, not replaced |
| Broad refactor of all large side-effect-heavy modules | Only targeted extractions needed for safer live-refresh work belong in scope |
| Authentication and authorization hardening | Important, but outside this milestone's performance/resilience goal |
| Vendor expansion beyond current focus | Deferred until the live-refresh path is stable |
