## Phase Details

### Phase 38: State Engine
**Goal**: Backend holds a single source of truth for all live device state -- health, severity, metrics, and staleness -- and emits only meaningful changes to the WebSocket delta layer
**Depends on**: Nothing (first phase of v1.5.3; architectural centerpiece with zero new-package dependencies)
**Requirements**: STATE-01, STATE-02, STATE-03, STATE-04, STATE-05
**Success Criteria** (what must be TRUE):
  1. Backend maintains a thread-safe in-memory store keyed by device UUID that holds per-device status, metrics, health enum, severity per metric, staleness, and failure counts -- accessible by any goroutine without data races (CI enforces `-race`)
  2. Each device's health state is a single enum (healthy/warning/critical/unknown) computed by the backend from per-metric severity evaluations using hardcoded thresholds with hysteresis (CPU warn 70%/clear 60%, critical 90%/clear 80%; same pattern for memory, temperature)
  3. A single SNMP timeout sets the device to soft-down (degraded); only after 3 consecutive failures does the device transition to hard-down -- and a single successful poll immediately recovers it
  4. When the state engine receives an update, it computes a diff against previous state and only emits changes to the existing FNV-64a delta WebSocket broadcast layer -- unchanged devices produce no broadcast
  5. Devices that have not been polled within 2x their expected interval are marked stale by the broadcast tick, with a last_polled_at timestamp available for downstream consumers
**Plans:** 2/2 plans complete
- [x] 38-01-PLAN.md -- Type foundation + threshold evaluation with hysteresis (internal/state/store.go types, health.go, health_test.go)
- [x] 38-02-PLAN.md -- Store methods: Update/Snapshot/Changes/Start/Stop/staleness tick + comprehensive race-safe tests

### Phase 39: Domain Types & DB Migration
**Goal**: The domain layer and database schema support OID volatility classification and per-device polling frequency so that the scheduler and collectors have typed foundations to build on
**Depends on**: Phase 38 (uses VolatilityClass and HealthStatus types defined in state engine)
**Requirements**: POLL-03, POLL-05
**Success Criteria** (what must be TRUE):
  1. Vendor config YAML defines OID groups classified into three volatility tiers (static, operational, performance) and the vendor registry resolves the correct OIDs per tier per vendor
  2. Devices are auto-classified into one of three poll classes based on
     device_type, with the following performance-tier intervals:
     - **core** = 30s (router, switch)
     - **standard** = 60s (ap, unknown)
     - **low** = 300s (virtual)
     Operators may set a per-device `poll_interval_override` (seconds, performance
     tier only) to bypass the class default. Operational-class polling
     (sysUpTime, ifOperStatus) defaults to 60s system-wide; static-class
     polling defaults to 300s system-wide. Both are system-defined in v1.5.3
     and become configurable in a later milestone.
  3. SQLite migration adds `poll_class` and `poll_interval_override` columns to the devices table, and existing devices receive correct auto-classification on migration
**Plans**: 4 plans
Plans:
- [x] 39-01-PLAN.md — Domain types: PollClass, VolatilityClass, interval constants, ClassifyPollClass helper, Device fields
- [x] 39-02-PLAN.md — Vendor schema: nested SNMPConfig (static/operational/performance), per-tier registry resolvers, YAML migration, PollDeviceMetrics consumer fix
- [x] 39-03-PLAN.md — DB migration 000016 + Go-level poll_class backfill + DeviceRepo CRUD wiring for new columns
- [x] 39-04-PLAN.md — probeDevice auto-reclassify with override guard + ROADMAP success criterion 2 update

### Phase 40: Collectors
**Goal**: Stateless collector functions exist for each volatility class, producing typed results that the state engine can consume -- making data collection independent of scheduling and state management
**Depends on**: Phase 39 (needs volatility class definitions and vendor OID mappings)
**Requirements**: PIPE-01, PIPE-02, PIPE-04
**Success Criteria** (what must be TRUE):
  1. PerformanceCollector polls CPU, memory, temperature, and interface counters via SNMP as the primary data source; Prometheus is used only as optional enrichment when configured and reachable
  2. OperationalCollector and StaticCollector exist as separate functions wrapping existing SNMP client calls, each producing typed result structs that implement a common StateUpdate interface
  3. Counter rate computation correctly handles resets (new < old = discard sample), enforces sanity bounds (rate > 110% link speed = discard), and skips rate on first poll after a gap -- verified by unit tests covering all edge cases
**Plans**: 4 plans

- [x] 40-02-PLAN.md -- PerformanceCollector plus legacy worker seam extraction for link matching and PIPE-04 rate correctness
- [x] 40-03-PLAN.md -- OperationalCollector and StaticCollector wrappers over SNMP operational/discovery primitives
- [x] 40-04-PLAN.md -- Prometheus enrichment collector with hostname/probe/alert-only scope

### Phase 41: Jittered Scheduler
**Goal**: Polls are distributed across the interval window with per-device timing offsets and concurrency limits, eliminating thundering herd bursts and separating discovery walks from metrics polling
**Depends on**: Phase 39 (needs PollClass and device type classification), Phase 40 (scheduler produces PollTasks consumed by collectors)
**Requirements**: POLL-01, POLL-02, POLL-04
**Success Criteria** (what must be TRUE):
  1. Discovery walks (ifTable, ifXTable, LLDP, CDP) run on a slower shared static cadence (`300s` system-wide in v1.5.3) completely independent of performance metrics polling
  2. Performance metrics polling runs per device, with the interval determined by the device's poll class (`core=30s`, `standard=60s`, `low=300s`) plus a performance-only override
  3. Each device's initial poll offset is deterministically computed from FNV hash of its UUID, spreading polls evenly across the interval window -- verified by distribution tests showing no burst clustering
  4. A concurrency limiter caps simultaneous SNMP operations, preventing network saturation regardless of how many device timers fire in the same window
**Plans**: 1 plan
Plans:
- [x] 47-01-PLAN.md — Close the final planning traceability drift before milestone archival

### Phase 42: Pipeline Orchestrator & Cutover
**Goal**: A unified PipelineOrchestrator replaces both Poller and MetricsCollector, wiring the scheduler to worker goroutines to the state engine to WebSocket broadcast -- completing the architectural transition
**Depends on**: Phase 38, Phase 40, Phase 41 (wires together state engine, collectors, and scheduler)
**Requirements**: PIPE-03
**Success Criteria** (what must be TRUE):
  1. The old Poller and MetricsCollector are fully replaced -- no dual-pipeline running; `main.go` wires the new PipelineOrchestrator as the sole polling and metrics delivery mechanism
  2. Worker pool goroutines consume PollTasks from the scheduler, dispatch to the correct collector by volatility class, and feed results into the state engine -- end-to-end data flow is functional
  3. Broadcast goroutine builds snapshots from the state store on a fixed tick (not triggered by individual polls), and existing WebSocket clients receive live metrics without regression from the old system
  4. The application starts, polls devices on their classified schedules, computes health state, and broadcasts deltas to connected frontends -- observable as a working topology map with live stats
**Plans**: 4/4 plans complete

### Phase 43: WebSocket Detail-on-Demand
**Goal**: Users can click a device on the canvas to subscribe to higher-frequency detail updates for that device, receiving richer data without increasing broadcast load for all clients
**Depends on**: Phase 42 (requires working pipeline and state store)
**Requirements**: WS-02
**Success Criteria** (what must be TRUE):
  1. Frontend sends a `subscribe_detail` message with a device ID over the existing WebSocket connection, and the backend tracks this subscription per client
  2. Subscribed clients receive higher-frequency detail updates for the selected device (pushed after each poll cycle for that device) while other clients continue receiving only the regular broadcast
  3. Deselecting the device (or closing the panel) sends `unsubscribe_detail`, and the backend immediately stops sending detail updates for that device to that client
