# Phase 04.1: Manually Setup Links Using Detected Ports from SNMP - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can manually create, edit, and delete network links between devices by selecting from SNMP-detected interfaces/ports. LLDP/CDP neighbors are auto-created as links on device probe. All links persist to the backend database, replacing the current localStorage-only approach.

</domain>

<decisions>
## Implementation Decisions

### Port/Interface Display
- Filtered list: only show meaningful interfaces (exclude loopback, null, management)
- Display per interface: name, speed, oper status
- DOWN interfaces shown at bottom, dimmed, but still selectable (for pre-wiring)
- Ports already used in another link are selectable but show a warning annotation (e.g., "in use by link to switch-dist")

### Link Creation UX
- Side panel dialog for link creation (reuse existing SidePanel component)
- Form: pick source device + source port dropdown, then target device + target port dropdown
- Port dropdowns show filtered interface list with name, speed, and status

### LLDP Auto-Discovery
- Auto-create links when LLDP/CDP neighbors are found during device probe
- Only runs on device add or manual re-probe (not on every polling cycle)
- Both devices must exist in the system for auto-link creation
- Upsert silently on re-probe: update if port names changed, no-op if identical
- No visual distinction between auto-discovered and manual links on canvas
- discovery_protocol stored in data model for filtering/debugging only

### Link Management
- Click link edge → opens SidePanel with link details (source/target device+port, protocol, speed)
- Edit allows changing ports only (same devices); to change devices, delete and recreate
- Delete requires simple confirmation prompt
- Existing localStorage manual edges migrated to backend on first load, then localStorage cleared

### Claude's Discretion
- Link creation trigger (toolbar button, keyboard shortcut, or context menu entry)
- Interface filtering logic (which interface types to exclude)
- Migration error handling for localStorage edges
- Side panel layout and styling details

</decisions>

<specifics>
## Specific Ideas

- Port selection preview style matches the filtered list mockup: `● ether1  1Gbps  UP` with dots for status
- Link details panel shows the bidirectional relationship clearly: `router-core:ether1 ↔ switch-dist:ether24`
- Auto-discovery feedback: show what was found and created during probe (neighbor count, links created, unresolved neighbors)

</specifics>

## Existing Code Insights

### Reusable Assets
- `SidePanel` component (`frontend/src/components/SidePanel.tsx`): Slide-out panel, can be extended for link creation/details views
- `ContextMenu` component (`frontend/src/components/ContextMenu.tsx`): Viewport-aware positioning, can add "Create Link" entry
- `LinkEdge` component (`frontend/src/components/LinkEdge.tsx`): Already renders links with utilization coloring and throughput labels
- Link domain model (`internal/domain/link.go`): Complete with `discovery_protocol` field supporting "lldp", "cdp", "manual"
- Link SQLite repository (`internal/repository/sqlite/link_repo.go`): Full CRUD + Upsert on (source_device, source_if, target_device, target_if) tuple

### Established Patterns
- API client pattern (`frontend/src/api/client.ts`): fetchLinks() exists; extend with createLink(), updateLink(), deleteLink()
- React Flow edge handling: Canvas already builds edges from both backend links (buildTopologyEdges) and manual edges (buildManualEdges)
- SNMP discovery (`internal/snmp/discovery.go`): DiscoverDevice returns `[]NeighborInfo` with RemoteChassisID, RemotePortID, RemoteSysName, LocalIfName, Protocol

### Integration Points
- Backend needs: POST /api/v1/links, PUT /api/v1/links/:id, DELETE /api/v1/links/:id, GET /api/v1/devices/:id/interfaces
- Device service ProbeDevice needs to call link creation with discovered neighbors
- Canvas.tsx manual edge localStorage logic needs replacement with backend API calls
- Frontend needs interface data from backend (currently only available during SNMP discovery, not exposed via API)
