Filter out interfaces where if_name starts with "lo" (loopback), "Null", or is empty. Sort: oper_status "up" first, then by if_name alphabetically. In `internal/service/device_service.go`, extend the `probeDevice` method to auto-create links from discovered neighbors. Add this logic AFTER the device fields are updated and saved (after the `s.deviceRepo.Update(fresh)` call succeeds):

1. Iterate over `result.Neighbors` (which is `[]snmp.NeighborInfo`).
2. For each neighbor, resolve the remote device:
   - Look up by `RemoteSysName` using `s.deviceRepo.GetBySysName(remoteSysName)` — need to check if this method exists. If not, iterate `GetAll()` and match by SysName. Since this runs during probe (not hot path), iterating all devices is acceptable.
   - If no matching device found in the system, skip (per user decision: "Both devices must exist in the system for auto-link creation").
3. For each resolved neighbor, call `s.linkRepo.Upsert(&domain.Link{...})` with:
   - SourceDeviceID: probed device ID
   - SourceIfName: neighbor.LocalIfName
   - TargetDeviceID: resolved remote device ID
   - TargetIfName: neighbor.RemotePortID (this is the remote port name from LLDP/CDP)
   - DiscoveryProtocol: neighbor.Protocol
4. Log: `"Auto-linked %s:%s <-> %s:%s via %s"` for each created/updated link.
5. Log: `"Skipping neighbor %s: device not found in system"` for unresolved neighbors.

This upsert behavior means re-probing silently updates existing links if port names changed, and creates new ones if not found — matching the user decision for upsert on re-probe.

Also add a `GetBySysName(sysName string) (*domain.Device, error)` method to `domain.DeviceRepository` interface and implement it in `internal/repository/sqlite/device_repo.go` — simple SELECT WHERE sys_name = ? LIMIT 1.
