# Roadmap: Theia v1.3.0 Frontend Redesign

## Overview

Transform Theia's frontend from its current utilitarian dark UI into the Neon Topography design system with dual-theme support and OSPF area-based navigation. The work flows from a CSS token foundation (which gates everything), through systematic component restyling and parallel area backend buildout, culminating in the Area Hub view and filtered topology canvas. Four phases, strict dependency chain: tokens first, then visual and data work in parallel, then the capstone area experience.

## Phase Details

### Phase 1: Design Token Foundation and Theme Infrastructure
**Goal**: Users can switch between visually complete dark and light themes, with the entire color system driven by CSS tokens and both fonts rendering correctly
**Depends on**: Nothing (first phase)
**Requirements**: FOUND-01, FOUND-02, FOUND-03, FOUND-04, FOUND-05, FOUND-06, THEME-01, THEME-02, THEME-03, THEME-04
**Success Criteria** (what must be TRUE):
  1. User can toggle between dark and light themes via a UI control and the entire app responds immediately (no stale colors)
  2. User's theme choice persists across browser sessions -- refreshing the page keeps the selected theme with no flash of wrong theme
  3. A new user with no saved preference sees the theme matching their OS dark/light setting
  4. All text in the app renders in Outfit (display/UI) or JetBrains Mono (metrics/code) -- no fallback system fonts visible
  5. No hardcoded hex color values remain in any component -- all colors respond to theme changes
**Plans**: 3 plans

Plans:

- [x] 01-02-PLAN.md — ThemeProvider context, ReactFlow v12 import migration, colorMode wiring, and NavBar theme toggle
- [x] 01-03-PLAN.md — Replace all hardcoded hex colors and old Tailwind class names across 8 component files
