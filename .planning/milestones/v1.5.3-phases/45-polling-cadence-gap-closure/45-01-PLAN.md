<objective>
Close `INT-01` by adding a narrow, immediate scheduler re-due path for
poll-override changes while preserving Phase 41's periodic, pull-based device
inventory model.

Purpose:
- Satisfy roadmap success criteria 1 and 2 for Phase 45 without reopening the
  broader scheduler architecture.
- Make the API/UI promise from Phase 44 true in runtime behavior: an override
  save affects the next performance poll cycle for that device.
- Keep the fix scoped to the affected device's performance task rather than
  turning CRUD updates into a general push-sync model.

Output:
- `Scheduler.ReduePerformanceTask(...)` (or an equivalently named exported
  scheduler hook) re-dueing only the affected device's performance task.
- `DeviceService.UpdateDevice(...)` triggering that hook only when
  `poll_interval_override` actually changes.
- Bootstrap wiring in `cmd/theia/main.go` plus service/scheduler regression
  coverage proving both the next emitted performance task uses the updated
  override cadence and the production startup path attaches `sched` to
  `DeviceService`.
</objective>

    Extend `internal/scheduler/scheduler.go` with an exported targeted command
    path named `ReduePerformanceTask(device domain.Device, changedAt time.Time)`
    and keep all scheduler state ownership inside the scheduler goroutine.

    Implement it exactly this way:

    1. Add a dedicated buffered command channel plus a private request struct
       for targeted performance re-due operations. Consume that channel inside
       the existing `run(...)` select loop; do not mutate `s.items`, `s.heap`,
       or `s.ready` directly from external goroutines.

    2. `ReduePerformanceTask(...)` must:
       - return immediately when the scheduler is stopped
       - normalize `changedAt` to `UTC()`
       - enqueue only one request carrying the updated `domain.Device`

    3. When the run loop handles that request, update only
       `TaskKey(device.ID, domain.VolatilityClassPerformance)`:
       - recompute the interval with
         `EffectiveInterval(device, domain.VolatilityClassPerformance)`
       - write the updated `device`, `PollClass`, `ExpectedInterval`, and
         `interval` back onto the heap item
       - set `dueAt` and `task.DueAt` to `changedAt`

    4. Handle all scheduler states explicitly:
       - heap item present in `s.heap`: update `dueAt` and call `heap.Fix(...)`
       - queued in `s.ready`: update fields, remove it from its current queue
         position, then push it to the front of the performance queue
       - in flight: set `pending = true`, update `dueAt` / `interval`, and let
         `handleCompletion(...)` enqueue the immediate rerun
       - missing but `device.Managed == true`: create exactly one performance
         item and enqueue it immediately
       - unmanaged device: ignore the request

    5. Do not touch operational or static items. Do not call
       `refreshDevices(...)` from this path. Do not add a generic CRUD ->
       scheduler push sync.

    6. Add scheduler tests covering these exact scenarios:
       - existing heap item becomes immediately due with the recomputed override
         interval
       - in-flight performance item becomes one pending immediate rerun with the
         new interval
       - missing managed device creates only the performance item, not all three
         volatility items
       - unmanaged device is ignored
  </action>

</threat_model>
